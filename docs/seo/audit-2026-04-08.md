# SEO Audit — 2026-04-08

On-page audit across `pw2d.com` and `coffee2decide.com` (the two live tenants).
No authenticated tooling (GSC/GA4/PostHog) was available — purely on-page HTML + robots + sitemap + code cross-reference.

## TL;DR

The SEO layer was built for a single-brand site and then multi-tenancy was layered on top. As a result, **every non-central tenant is indexing under "pw2d" branding** despite live tenant domains. The tenant-facing catastrophe lives in three files — nothing deep or architectural, but the effect is large.

**The three bleeding wounds:**

| # | Finding | Files | Severity |
|---|---|---|---|
| H1 | Hardcoded `pw2d` brand in titles + `og:site_name` on every tenant | [SeoSchema.php](../../app/Support/SeoSchema.php), [app.blade.php](../../resources/views/components/layouts/app.blade.php) | **HIGH** |
| H2 | `public/robots.txt` is a static file pointing all tenants at `pw2d.com/sitemap.xml` | [robots.txt](../../public/robots.txt) | **HIGH** |
| H3 | Homepage (`Home.php`) sets no meta at all — falls through to pw2d defaults | [Home.php](../../app/Livewire/Home.php) | **HIGH** |

All three can be fixed in one small PR before anything else. See [Fix Plan](#fix-plan) below.

## Findings

### H1 — Cross-tenant brand bleed in meta/OG/title tags (HIGH)

**Evidence from coffee2decide.com (live):**

| Page | `<title>` observed | Should be |
|---|---|---|
| `/` | `pw2d - Power to Decide \| AI Tech Recommendations` | Tenant-specific, e.g. `Coffee2Decide — Find Your Perfect Espresso Machine` |
| `/compare/espresso-machines-grinders` | `Espresso Machines & Grinders - Browse Categories \| pw2d` | `… \| Coffee2Decide` |
| `/compare/semi-automatic-manual-espresso-machines` | `Semi-Automatic & Manual Espresso Machines - Compare Best Models in 2026 \| pw2d` | `… \| Coffee2Decide` |
| `/compare/super-automatic-espresso-machines?preset=latte-lover` | `Best Super-Automatic Espresso Machines for Latte Lover \| pw2d` | `… \| Coffee2Decide` |

Every page also renders the same hardcoded OG site name:

```html
<meta property="og:site_name" content="pw2d - Power to Decide">
```

**Root causes (3 files):**

1. [app/Support/SeoSchema.php](../../app/Support/SeoSchema.php) — hardcoded `| pw2d` string suffixes:
   - Line 126: `"{$category->name} - Browse Categories | pw2d"` (parent category)
   - Line 145: `"{$category->name} - Compare Best Models in {$currentYear} | pw2d"` (leaf)
   - Line 166: `"Best {$category->name} for {$activePreset->name} | pw2d"` (preset)

2. [resources/views/components/layouts/app.blade.php](../../resources/views/components/layouts/app.blade.php):
   - Line 9: `$metaTitle ?? 'pw2d - Power to Decide | AI Tech Recommendations'`
   - Line 10–11: default description is the generic tech copy
   - Line 15: same hardcoded fallback title
   - Line 19: `$defaultImage = $ogImage ?? asset('images/logo.webp')` — pw2d's logo, not tenant's
   - Line 24: `<meta property="og:site_name" content="pw2d - Power to Decide">` — hardcoded

3. The tenant model already has `brand_name` available via `tenant('brand_name')` (used correctly in the footer at line 141/154/177). It's simply not used in the head.

**Impact:** Google will index `coffee2decide.com` pages as `pw2d`-branded. Every branded search query for "Coffee2Decide" or a tenant-specific brand will either not find results, or find results labelled under the wrong brand, undermining the multi-tenant value proposition entirely.

**Fix shape:** Thread `tenant('brand_name')` through `SeoSchema` and the layout. No new abstraction needed.

### H2 — robots.txt points all tenants at pw2d.com's sitemap (HIGH)

**Evidence:**
```bash
$ curl -s https://pw2d.com/robots.txt
User-agent: *
Disallow: /admin/
Disallow: /livewire/

Sitemap: https://pw2d.com/sitemap.xml

$ curl -s https://coffee2decide.com/robots.txt
# (identical to above)
```

**Root cause:** [public/robots.txt](../../public/robots.txt) is a static file served by nginx, identical regardless of `Host` header. It always advertises `https://pw2d.com/sitemap.xml` as the sitemap.

**Impact:**
- **Google does not ingest cross-domain sitemaps** unless both domains are verified in the same GSC account *and* the sitemap is submitted as a cross-domain property. This does not happen automatically. So `coffee2decide.com` product pages are not being discovered via sitemap — they rely entirely on internal link crawling.
- Each tenant loses a primary discovery channel.

**Fix shape:** Replace the static file with a route (`Route::get('/robots.txt', …)`) that renders `Sitemap: https://{current host}/sitemap.xml`. Delete `public/robots.txt`. Tiny change, ~10 LOC.

**Note:** also verified `https://pw2d.com/sitemap.xml` returns **HTTP 200** with 1,050+ URLs, despite `SitemapController` having `abort(404)` when tenancy isn't initialized. This implies `pw2d.com` is actually being resolved as a *tenant* (not a central domain) in production, despite being listed in `config/tenancy.php` line 17 `central_domains`. Either `APP_CENTRAL_DOMAIN` env is different on prod, or a separate admin subdomain exists. **Worth a quick config review** — the mismatch between dev and prod tenancy resolution is a foot-gun.

### H3 — Homepage sets no meta at all (HIGH)

**Evidence:** `Home.php::render()` returns a view without passing any `metaTitle`, `metaDescription`, `canonicalUrl`, `ogImage`, or `schemaJson`. The homepage for every tenant thus falls through to the hardcoded defaults in `app.blade.php` line 9–19.

**Impact:** On `coffee2decide.com/`:
- `<title>` = "pw2d - Power to Decide | AI Tech Recommendations"
- `<meta name="description">` = "Discover the best tech products…"
- `og:image` = `https://coffee2decide.com/images/logo.webp` (the pw2d logo, served from the tenant domain)

Interestingly, the H1 *is* tenant-specific ("Your Personal Coffee Sommelier"), which means tenant customization exists somewhere (Blade template or tenant data). It just never made it into the `<head>`.

**Fix shape:** Add a `forHomepage()` method to `SeoSchema` that reads `tenant('brand_name')`, `tenant('seo_title')`, `tenant('seo_description')` and returns the meta bundle. Then wire it through `Home.php::render()` the same way `ProductCompare.php` does.

Requires a handful of tenant fields to be added (or decided on): `seo_title`, `seo_description`, `seo_default_image`. These can live in the `tenants.data` JSON so no migration is needed.

### M1 — No analytics on tenants other than pw2d (MEDIUM)

**Evidence:** On `coffee2decide.com/` the HTML head has **no** PostHog init script, **no** `gtag` script, and **no** `google-site-verification` meta. On `pw2d.com/` all three are present (PostHog key `phc_b5uhGFuGb7jiBaRF…`, GA4 `G-KJRBQB79H9`).

This is by design — `app.blade.php` lines 48–51 read these from the **tenant-scoped** `Setting::get()` store, and the coffee2decide tenant has no rows. So the tenant is effectively invisible to all three tools.

**Impact:** This is independent of the on-page issues — it means **you cannot measure** SEO for `coffee2decide.com` right now. Any SEO fix shipped to that tenant cannot be validated. Fixing this is a prerequisite for the "SEO integrations" spec (coming next).

**Fix shape:**
1. Short term: a Filament admin page to edit these 4 settings per tenant.
2. Medium term: spec 014 (SEO monitoring integration) — design to follow.

### M2 — `og:image` is always the logo on category/preset pages (MEDIUM)

**Evidence:** `SeoSchema::forParentCategory()` and `forLeafCategory()` return `'ogImage' => null`. The layout then falls through to `asset('images/logo.webp')`. Only `forSelectedProduct()` sets a real image.

**Impact:** Social shares of category URLs ("best espresso machines") show a brand logo instead of, say, the hero product image. Measurably lower CTR on shared links vs. image-rich previews.

**Fix shape:** In `forLeafCategory()` + `forParentCategory()`, use the top-scored product's image (or the category `image` column if set) as `ogImage`.

---

### M3 — Sitemap omits static pages (MEDIUM)

[SitemapController](../../app/Http/Controllers/SitemapController.php) emits home + categories + products + presets, but not `/about`, `/contact`, `/privacy-policy`, `/terms-of-service`. These routes exist in `routes/web.php` lines 33–36. Google will eventually crawl them but explicit inclusion is cheap and correct.

---

### M4 — Product schema omits `offers` by design (MEDIUM, revisit)

[SeoSchema.php:231](../../app/Support/SeoSchema.php#L231) comment:
> Offers (price) intentionally omitted — scraped prices are estimates and violate Google's strict price-matching rules for Merchant Center rich snippets.

Defensible, but worth revisiting: you lose price rich-snippet eligibility. If `ProductOffer` staleness can be held under Google's threshold (typically ~2 days) by the existing scraping cadence, you could emit `offers` only when `offer.updated_at > now() - 2 days`. Separate decision, not in scope for this audit fix.

---

### M5 — Parent category schema is thin (MEDIUM)

`CollectionPage` schema has only `name` + `description`. Missing:
- `hasPart` array referencing subcategories (useful entity graph)
- `breadcrumb` property
- `url`

Low-risk, moderate upside. Bundle with H1 fix.

### Lower-priority observations

- **L1** — `twitter:site` and `twitter:creator` handles not set. Minor.
- **L2** — Sitemap response headers include `Cache-Control: no-cache, private` — wasteful to rebuild on every request for a file that changes hourly at most. Add a 10-minute page cache.
- **L3** — No `<meta name="robots" content="noindex">` on empty leaf categories. A category with zero products still emits an ItemList schema with an empty `itemListElement` — soft-404 risk.
- **L4** — `<html lang>` uses `app()->getLocale()`, not tenant locale. OK today (single locale), but will break when a tenant wants a different language.
- **L5** — Preset page URLs use `?preset=slug` query string. This is fine (preset pages are legitimately distinct content), but consider `/compare/{category}/for/{preset}` as a cleaner path if other SEO work is happening. Purely aesthetic.
- **L6** — Canonical for `/product/{slug}` is stable, good. No `noindex` logic when product is hidden (`is_ignored=true`) — but those products are already excluded from the sitemap, so exposure is only via direct links. Low risk.

## Fix Plan

Two phases. Phase 1 is a single small PR that stops the bleeding.

### Phase 1 — Stop Brand Bleed (target: this week)

One spec, one PR:

1. **Add tenant SEO fields** to `tenants.data` JSON:
   - `seo_title` (fallback used by homepage + as suffix replacing "| pw2d")
   - `seo_description` (homepage meta description)
   - `seo_default_image` (fallback og:image when no product image available)
   - `brand_name` (already exists per footer usage)

2. **Refactor SeoSchema.php** to use `tenant('brand_name')` / `tenant('seo_title')` instead of hardcoded `pw2d`. Add `forHomepage()` method. Set `ogImage` to top product image for category/preset pages. Add `hasPart` to parent category schema. Include static pages in sitemap.

3. **Refactor app.blade.php** head to read tenant-scoped defaults:
   - `og:site_name` → `tenant('brand_name') ?? config('app.name')`
   - Default title → `tenant('seo_title')`
   - Default description → `tenant('seo_description')`
   - Default og:image → `tenant('seo_default_image') ?? asset('images/logo.webp')`

4. **Wire Home.php::render()** to call `SeoSchema::forHomepage()` and pass meta to the view (same pattern as `ProductCompare.php` lines 497–522).

5. **Replace `public/robots.txt`** with a route that renders per-tenant sitemap URL based on `request()->getSchemeAndHttpHost()`. Delete the static file.

6. **Seed coffee2decide tenant settings**: `posthog_key`, `ga_measurement_id`, `google_site_verification`. (Quick Filament admin page or direct tinker seed — whichever is faster.)

7. **Tests** (Pest + `InteractsWithTenancy` pattern): for each of the 4 page scenarios (home / parent / leaf / preset / product), assert the `<title>` contains `tenant('brand_name')` and NOT `pw2d` when running under a non-pw2d tenant.

**Estimated scope:** ~15 files touched, ~300 LOC, mostly trivial substitutions.

### Phase 2 — SEO Monitoring Integration (separate spec)

Design to follow in `docs/specs/014-seo-monitoring-integration.md`. Scope:
- `GoogleSearchConsoleService` (service account, per-tenant site URL)
- `GoogleAnalyticsService` (GA4 Data API)
- `PosthogService` (per-tenant project key)
- `seo_metrics` table with weekly aggregates per (tenant, url, date)
- Filament dashboard page showing query impressions / CTR / position trends
- Scheduled Artisan command pulling data nightly

This gives the team visibility to validate that Phase 1 actually moved the needle.
